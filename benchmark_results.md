## Verdict: GO (with a scaling caveat)

**The runtime CAST does NOT add a major regression.** Holding the query skeleton and
result selectivity identical (`CAST = ` vs string `=`, same 85 / 882 / 9,105 rows
returned), the CAST-to-float adds a **stable ~2.2x** overhead on the value comparison:
~2.3x @ 10k, ~2.2x @ 100k, ~2.2x @ 1M. The ratio is flat across two orders of magnitude,
so this is a **bounded constant factor, not a super-linear cliff**.

Crucially, **both** the existing string-equality filter and the CAST filter are already
`O(N)` full scans of the `assessments` table — there is no usable index on the JSON
`value` column for either. The CAST therefore does not change the complexity class or
introduce a _new_ scaling problem; it makes an already-linear comparison ~2x heavier.
On realistic range queries the CAST's absolute latency stays in the same ballpark as the
equality baseline that is already in production (e.g. @100k: baseline-eq ~142 ms vs
CAST `>` ~100 ms — the CAST `>` is even faster here because it returns a different row
set, which is why the same-selectivity table above is the decisive comparison).

**Recommendation: GO** — ship the CAST approach in phase 2. The overhead is acceptable.

**Caveat / future work (orthogonal to this decision):** if assessment-filter latency at
scale becomes a hard product requirement, neither approach meets a sub-100 ms bar at
100k+ (string-eq is already ~140 ms). The fix is a dedicated **generated/indexed numeric
column** on `assessments` (populated when the JSON value parses as a number). That would
break the `O(N)` scan for _both_ string and numeric filters, so it is a separate,
broadly-beneficial optimization — not a reason to block the CAST.

### Methodology caveats

- SQLite only. MySQL/Postgres/MSSQL `CAST(... AS FLOAT/REAL)` and `substring` semantics
  differ; the prototype's dialect branches (`_get_sql_json_comparison_func`) are not
  exercised here. Re-run on the production dialect before final sign-off.
- Seeding uses **bulk inserts into the real ORM tables** (`SqlTraceInfo` / `SqlAssessments`)
  with the same JSON value encoding the store APIs produce — public `log_trace` +
  `create_assessment` are too slow at 100k+. The BASELINE row is the **real**
  `store.search_traces` path; `CONSTRUCTED eq` reproduces the store's subquery+join
  skeleton and is validated to return identical rows (matching row counts), so the CAST
  delta is measured against a faithful skeleton.
- The `ATTR-NUMERIC` reference filters an indexed numeric column but at ~80% selectivity,
  so it reflects scan+materialize cost, not an index-seek speedup; treat it as a same-cost
  reference point, not as the speedup a _selective_ indexed numeric column would give.
