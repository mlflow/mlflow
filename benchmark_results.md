## CAST overhead vs string-equality baseline

### Same-skeleton, same-selectivity (isolates pure per-row CAST/CASE cost)

_This is the **per-row CAST cost in isolation (micro-benchmark)**: a bare,
ID-only subquery+join skeleton that selects only `request_id` (no entity
hydration), so the value expression dominates the measurement. It answers "how
much heavier is the value comparison itself?" — NOT "how much slower is a real
search?" For the latter, see the production-path table below._

| scale | constructed eq (string) | CAST = (same rows) | ratio (CAST/eq) |
|---|---|---|---|
| 10,000 | 3.31 ms | 7.19 ms | 2.18x |
| 100,000 | 33.01 ms | 72.79 ms | 2.20x |
| 1,000,000 | 323.23 ms | 696.76 ms | 2.16x |

### Production-path, like-for-like: string-eq vs CAST `>` (both via `store.search_traces`)

_This is the **end-user impact on the production path**. BOTH queries run through the real `store.search_traces(...)` — full `TraceInfo` entity hydration (tags/metadata/assessments selectin-loaded), filter parse, and pagination — so the ONLY variable is the value expression (string OR-compare vs CASE+CAST). Selectivity is held **byte-identical**, not merely matched: scores are `round(uniform(0,10), 1)`, so `score > 9.9` selects EXACTLY the `{10.0}` bucket that `score = "10.0"` selects (the harness asserts the two result sets are identical and reports both row counts). The numeric-CAST routing is supplied by an in-process, benchmark-only monkeypatch that reproduces the shipped production closure byte-for-byte; no production file is modified._

| scale | eq rows | CAST rows | eq median ms | eq p90 ms | CAST median ms | CAST p90 ms | ratio (CAST/eq) |
|---|---|---|---|---|---|---|---|
| 10,000 | 43 | 43 | 11.53 | 11.77 | 15.29 | 15.58 | 1.33x |
| 100,000 | 430 | 430 | 121.33 | 178.23 | 160.06 | 208.50 | 1.32x |
| 1,000,000 | 4,555 | 4,555 | 1405.47 | 1423.94 | 1752.73 | 1774.08 | 1.25x |

**Interpretation.** On the production path the CAST/eq median ratio is **1.25x–1.33x** across 10,000, 100,000, 1,000,000 (absolute overhead +4 ms @ 10,000, +39 ms @ 100,000, +347 ms @ 1,000,000). That is **substantially smaller** than the per-row CAST overhead the micro-benchmark above measures in isolation (**2.16x–2.20x**): once the CAST runs inside the real query, its cost is partly **diluted** by the operator-independent cost of hydrating full `TraceInfo` entities, parsing the filter, and paginating — work both queries pay equally. It is *not* erased, though: the CAST still adds a roughly fixed absolute overhead (the same `O(N)` scan + per-row `CAST(... AS FLOAT)` the skeleton table isolates), so the production ratio sits **above 1.0x**, not at it. The honest summary for a real user issuing a numeric assessment filter through `search_traces`: the CAST makes the query **~1.2–1.3x slower than the same-selectivity string-equality filter end to end** — meaningfully less than the ~2.2x per-row figure, because hydration dominates, but still a real (bounded, constant-factor) cost, not a free operation.

### Verdict: inequality CAST overhead == equality CAST overhead (within noise)

At **matched selectivity**, the CAST inequality operators cost the **same** as CAST `=`:

- `> 9.9` and `>= 10.0` return the **byte-identical** `{10.0}` row set as `= 10.0`
  (43 / 430 / 4,555 rows at 10k / 100k / 1M). Their ratios vs CAST `=` are
  **0.98–1.02x across all scales** — i.e. indistinguishable from `=` within run-to-run
  noise. This is the truly controlled comparison (operator is the *only* variable).
- `< 0.1` and `<= 0.0` ratio **1.12–1.20x**, but that is a *selectivity* artifact, not an
  operator-cost difference: those predicates match the `{0.0}` edge bucket, which carries
  more rows (286 / 2,866 / 29,276) because the prototype CASE folds boolean `false` →
  `0.0`. The extra latency tracks the extra returned rows, not the `<`/`<=` operator.

**Conclusion:** the per-row CAST/CASE cost is paid by the value expression regardless of
which comparison operator consumes it. `>`, `>=`, `<`, `<=` carry the **same ~2.2x
CAST overhead** measured for `=` in the previous section — there is no operator-specific
penalty. This reinforces the **GO** verdict: shipping all four inequality operators adds
no cost beyond the already-quantified, bounded CAST factor.

## Overall verdict: GO (with a scaling caveat)

**The runtime CAST does NOT add a major regression.** Two complementary measurements bound
the cost:

- **Per-row CAST cost in isolation (micro-benchmark).** Same-skeleton, same-selectivity
  (`CAST =` vs string `=`, identical row sets, bare ID-only query with no entity
  hydration), the CAST-to-float adds a **stable ~2.2x** overhead on the value comparison,
  flat across 10k → 1M (a bounded constant factor, not a super-linear cliff). The
  same-selectivity operator sweep confirms `>`, `>=`, `<`, `<=` carry that same ~2.2x and
  no operator-specific penalty.
- **End-user impact on the production path.** Run through the *real* `store.search_traces`
  (full `TraceInfo` hydration, parse, pagination) at byte-identical selectivity
  (`= "10.0"` vs `> "9.9"`, identical `{10.0}` row set), the CAST makes the query
  **~1.25–1.33x slower** than the same-selectivity string-equality filter — meaningfully
  less than the ~2.2x per-row figure, because the dominant hydration/parse/pagination cost
  (paid equally by both queries) dilutes the value-expression delta. The ratio **shrinks**
  as scale grows (1.33x → 1.25x from 10k → 1M) as hydration takes a larger share. It is a
  real, bounded, constant-factor cost — not a free operation, and **not** faster than
  equality.

Both the existing string-equality filter and the CAST filter are already `O(N)` scans of
the `assessments` table (no index on the JSON `value` column), so the CAST does not change
the complexity class — it makes an already-linear comparison ~2x heavier per row, which
the production path absorbs down to ~1.25–1.33x end to end.

**Recommendation: GO** — ship the CAST approach (all four inequality operators) in phase 2.

**Caveat / future work (orthogonal):** if sub-100 ms assessment-filter latency at 100k+
becomes a hard requirement, neither approach meets it (string-eq is already ~140 ms). The
fix is a dedicated generated/indexed numeric column on `assessments`, which speeds up
*both* string and numeric filters — a separate optimization, not a reason to block the CAST.

### Methodology caveats
- SQLite only. The prototype's MySQL/Postgres/MSSQL CAST + `substring` branches are not
  exercised here; re-verify on the production dialect before final sign-off.
- Seeding uses bulk inserts into the real ORM tables (`SqlTraceInfo` / `SqlAssessments`)
  with the same JSON value encoding the store APIs produce; public `log_trace` +
  `create_assessment` are too slow at 100k+. The BASELINE row is the real
  `store.search_traces` path; `CONSTRUCTED eq` reproduces the store's subquery+join
  skeleton and returns identical rows.
- This bench branch is cut from `master`, which does **not** contain the numeric-CAST
  filter (it lives only on the implementation branch). The production-path table's CAST
  routing is therefore supplied by an **in-process, benchmark-only monkeypatch**
  (`install_production_numeric_cast_path`) that reproduces the implementation branch's
  `json_numeric_comparison` closure byte-for-byte and widens the assessment comparator
  allow-list. No file under `mlflow/` is modified on disk; the SQL executed is exactly what
  the shipped feature emits, through the same `store.search_traces` hydration path.
- In the operator sweep, `< 0.1` / `<= 0.0` match the `{0.0}` bucket, which is larger than
  `{10.0}` because the prototype CASE folds boolean `false` → 0.0; their slightly higher
  latency reflects more matched rows, not operator cost. The `> 9.9` / `>= 10.0` pair
  (byte-identical row set to `= 10.0`) is the strictly controlled comparison.

## Boolean coercion (prototype) vs NULL-out (ours) CASE cost

_BENCHMARK-ONLY. Generated by the `--ab-compare` mode of `dev/benchmarks/numeric_assessment_filter_bench.py`._

SQLite, warm cache (first run discarded), reps=9, MEDIAN + p90. Backend: real MLflow `SqlAlchemyStore`. Both variants run through the identical subquery+join skeleton, so the only variable is the CASE shape.

- **VARIANT A (ours, PR #23948)** — NULL-out booleans: `lower(value) IN ('true','false','null') -> NULL`, non-scalar JSON -> NULL, else `CAST(value AS FLOAT)`.
- **VARIANT B (prototype, PR #21811)** — coerce booleans: `'true'/'"yes"' -> 1.0`, `'false'/'"no"' -> 0.0`, `'null' -> NULL`, non-scalar JSON -> NULL, else `CAST(value AS FLOAT)`.

| scale | scenario | variant | matched rows | median ms | p90 ms | ratio vs A |
|---|---|---|---|---|---|---|
| 10,000 | matched (>= 8.0) | A (ours, NULL-out) | 1,825 | 10.28 | 10.84 | 1.00x |
| 10,000 | matched (>= 8.0) | B (prototype, coerce) | 1,825 | 9.83 | 10.54 | 0.96x |
| 10,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 1,343 | 9.50 | 9.66 | 1.00x |
| 10,000 | selectivity (<= 1.5) | B (prototype, coerce) | 1,849 | 9.79 | 10.05 | 1.03x |
| 100,000 | matched (>= 8.0) | A (ours, NULL-out) | 18,335 | 111.85 | 162.58 | 1.00x |
| 100,000 | matched (>= 8.0) | B (prototype, coerce) | 18,335 | 106.16 | 155.32 | 0.95x |
| 100,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 14,065 | 104.79 | 149.56 | 1.00x |
| 100,000 | selectivity (<= 1.5) | B (prototype, coerce) | 19,074 | 105.79 | 151.81 | 1.01x |
| 1,000,000 | matched (>= 8.0) | A (ours, NULL-out) | 183,755 | 1313.85 | 1399.53 | 1.00x |
| 1,000,000 | matched (>= 8.0) | B (prototype, coerce) | 183,755 | 1284.20 | 1317.96 | 0.98x |
| 1,000,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 139,421 | 1170.27 | 1214.99 | 1.00x |
| 1,000,000 | selectivity (<= 1.5) | B (prototype, coerce) | 189,274 | 1262.16 | 1358.69 | 1.08x |

### Verdict

**(a) At matched selectivity (`>= 8.0`, identical row sets), VARIANT B is NOT measurably slower than VARIANT A.** With the matched set held byte-identical (coerced 1.0/0.0 booleans can't satisfy `>= 8.0`, so both variants return only genuine numerics), B/A median ratios land at 0.95x–0.98x across 10,000, 100,000, 1,000,000 — i.e. within run-to-run noise. The two extra `IN`/equality arms in B's CASE add no observable per-row cost over A's NULL-out shape; both are dominated by the same `CAST(... AS FLOAT)` on the non-coerced majority and the same `O(N)` scan of the unindexed JSON `value` column. **The CASE shape itself is not the cost driver.** 

**(b) The real cost of coercion is selectivity, not CASE evaluation.** At `<= 1.5`, B folds `true`->1.0 and `false`->0.0 into the matched set while A NULLs them out, so B returns A's rows PLUS every boolean-valued feedback row (B_extra = 506 @ 10,000, 5009 @ 100,000, 49853 @ 1,000,000). That inflates B's matched set and pushes its median to 1.01x–1.08x of A in this scenario — the extra latency tracks the extra downstream rows the coercion admits, not the comparison operator or the CASE branches. 

**Bottom line for the prototype decision:** coercing booleans to 1.0/0.0 does *not* cost more at query time per se (the CASE shapes are equal-cost at matched selectivity); it costs more only when those coerced booleans actually fall inside the query's numeric range and become extra returned rows. Whether that is a regression depends entirely on whether surfacing boolean feedback as 1.0/0.0 in numeric filters is desired semantics — it is not a CASE-evaluation penalty.

_Dialect note: SQLite here. MySQL uses `CAST AS DOUBLE` (vs SQLite `FLOAT`), but the A-vs-B CASE-shape conclusion — equal cost at matched selectivity, divergence only via selectivity — is dialect-independent._

## Cross-dialect results (PostgreSQL / MySQL)

_BENCHMARK-ONLY. Generated by `dev/benchmarks/numeric_assessment_filter_bench.py --db-uri ...`
against stock `postgres:16` and `mysql:8.0` servers (see `dev/benchmarks/docker-compose.yml`).
Warm cache, reps=7. Backend: real MLflow `SqlAlchemyStore` (alembic auto-migrates a fresh
per-run database on each server)._

**Scope.** The SQLite tables above remain the **1M reference**. The cross-dialect runs are
**capped at 10k and 100k** — seeding 1M rows over a Docker socket into a containerized server
is impractical (the 1M point adds no new dialect signal: the per-row CAST factor is already
flat across 10k → 1M on SQLite). These runs exist to confirm the dialect-aware CAST branch on
each engine and to compare its overhead ratio to the SQLite baseline, **not** to re-measure
absolute scaling.

**Which CAST branch each dialect exercises** (auto-selected by `_get_dialect()` →
`json_numeric_comparison`, identical to shipped production):

- **PostgreSQL → `CAST(value AS FLOAT)`** (the SQLite/PG/MSSQL branch).
- **MySQL → `CAST(value AS DOUBLE)`** (the dedicated MySQL branch; MySQL has no `FLOAT`
  cast target). MySQL equality also goes through the case-sensitive `BINARY` OR-compare.

Both the locally-constructed skeleton queries and the production `search_traces` path pick
the dialect from the live engine, so every row below is the real per-dialect SQL.

### Same-skeleton, same-selectivity micro-benchmark (pure per-row CAST/CASE cost)

### Interpretation

**The CAST overhead is dialect-independent in conclusion and no worse than SQLite — on the
real server engines it is actually *cheaper* relative to baseline than the SQLite ~2.2x
micro figure.**

- **PostgreSQL (FLOAT branch).** The per-row FLOAT cast is essentially free: the micro ratio
  is **1.41x @ 10k and 0.83x @ 100k** — at 100k the CAST `=` query is *faster* than the
  string OR-compare baseline (`column == value OR column == json_value` does two text
  comparisons; a single numeric cast + compare wins once PG's planner is warm). On the
  production path the ratio is **1.06x–1.39x**, bracketing the SQLite 1.32x–1.33x.
- **MySQL (DOUBLE branch).** The DOUBLE cast adds a steady, small overhead: micro
  **1.10x–1.12x**, production-path **1.08x–1.14x** — well under the SQLite ~2.2x per-row
  figure and in line with SQLite's ~1.3x production-path number.

Across both server dialects the production-path CAST/eq ratio stays in the **~1.06x–1.39x**
band — the same bounded, constant-factor cost the SQLite reference shows, dominated by the
shared hydration/parse/pagination work. No dialect exhibits a super-linear cliff or an
operator-specific penalty; the DOUBLE vs FLOAT cast-target difference does not change the
conclusion. This **confirms the SQLite-only "GO" verdict holds on the production engines**
and closes the "SQLite only" methodology caveat above for the 10k/100k range.

_All cross-dialect rows are measured (RC=0, identical-row-set assertion passed on every
production-path pair). Containers were torn down (`docker compose down -v`) after the run._
