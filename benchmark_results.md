### Verdict: inequality CAST overhead == equality CAST overhead (within noise)

At **matched selectivity**, the CAST inequality operators cost the **same** as CAST `=`:

- `> 9.9` and `>= 10.0` return the **byte-identical** `{10.0}` row set as `= 10.0`
  (43 / 430 / 4,555 rows at 10k / 100k / 1M). Their ratios vs CAST `=` are
  **0.98–1.02x across all scales** — i.e. indistinguishable from `=` within run-to-run
  noise. This is the truly controlled comparison (operator is the *only* variable).
- `< 0.1` and `<= 0.0` ratio **1.05–1.15x**, but that is a *selectivity* artifact, not an
  operator-cost difference: those predicates match the `{0.0}` edge bucket, which carries
  more rows (286 / 2,866 / 29,276) because the prototype CASE folds boolean `false` →
  `0.0`. The extra latency tracks the extra returned rows, not the `<`/`<=` operator.

**Conclusion:** the per-row CAST/CASE cost is paid by the value expression regardless of
which comparison operator consumes it. `>`, `>=`, `<`, `<=` carry the **same ~2.2x
CAST overhead** measured for `=` in the previous section — there is no operator-specific
penalty. This reinforces the **GO** verdict: shipping all four inequality operators adds
no cost beyond the already-quantified, bounded CAST factor.

## Overall verdict: GO (with a scaling caveat)

**The runtime CAST does NOT add a major regression.** Same-skeleton, same-selectivity
(`CAST =` vs string `=`, identical row sets), the CAST-to-float adds a **stable ~2.2x**
overhead on the value comparison, flat across 10k → 1M (a bounded constant factor, not a
super-linear cliff). The same-selectivity operator sweep above confirms `>`, `>=`, `<`,
`<=` carry that same ~2.2x and no operator-specific penalty.

Both the existing string-equality filter and the CAST filter are already `O(N)` scans of
the `assessments` table (no index on the JSON `value` column), so the CAST does not change
the complexity class — it makes an already-linear comparison ~2x heavier. On realistic
range queries the CAST's absolute latency stays in the same ballpark as the equality
baseline already in production.

**Recommendation: GO** — ship the CAST approach (all four inequality operators) in phase 2.

**Caveat / future work (orthogonal):** if sub-100 ms assessment-filter latency at 100k+
becomes a hard requirement, neither approach meets it (string-eq is already ~140 ms). The
fix is a dedicated generated/indexed numeric column on `assessments`, which speeds up
*both* string and numeric filters — a separate optimization, not a reason to block the CAST.

### Methodology caveats
- SQLite only. The prototype's MySQL/Postgres/MSSQL CAST + `substring` branches are not
  exercised here; re-verify on the production dialect before final sign-off.
- Seeding uses bulk inserts into the real ORM tables (`SqlTraceInfo` / `SqlAssessments`)
  with the same JSON value encoding the store APIs produce; public `log_trace` +
  `create_assessment` are too slow at 100k+. The BASELINE row is the real
  `store.search_traces` path; `CONSTRUCTED eq` reproduces the store's subquery+join
  skeleton and returns identical rows.
- In the operator sweep, `< 0.1` / `<= 0.0` match the `{0.0}` bucket, which is larger than
  `{10.0}` because the prototype CASE folds boolean `false` → 0.0; their slightly higher
  latency reflects more matched rows, not operator cost. The `> 9.9` / `>= 10.0` pair
  (byte-identical row set to `= 10.0`) is the strictly controlled comparison.

## Boolean coercion (prototype) vs NULL-out (ours) CASE cost

_BENCHMARK-ONLY. Generated by the `--ab-compare` mode of `dev/benchmarks/numeric_assessment_filter_bench.py`._

SQLite, warm cache (first run discarded), reps=9, MEDIAN + p90. Backend: real MLflow `SqlAlchemyStore`. Both variants run through the identical subquery+join skeleton, so the only variable is the CASE shape.

- **VARIANT A (ours, PR #23948)** — NULL-out booleans: `lower(value) IN ('true','false','null') -> NULL`, non-scalar JSON -> NULL, else `CAST(value AS FLOAT)`.
- **VARIANT B (prototype, PR #21811)** — coerce booleans: `'true'/'"yes"' -> 1.0`, `'false'/'"no"' -> 0.0`, `'null' -> NULL`, non-scalar JSON -> NULL, else `CAST(value AS FLOAT)`.

| scale | scenario | variant | matched rows | median ms | p90 ms | ratio vs A |
|---|---|---|---|---|---|---|
| 10,000 | matched (>= 8.0) | A (ours, NULL-out) | 1,825 | 10.28 | 10.84 | 1.00x |
| 10,000 | matched (>= 8.0) | B (prototype, coerce) | 1,825 | 9.83 | 10.54 | 0.96x |
| 10,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 1,343 | 9.50 | 9.66 | 1.00x |
| 10,000 | selectivity (<= 1.5) | B (prototype, coerce) | 1,849 | 9.79 | 10.05 | 1.03x |
| 100,000 | matched (>= 8.0) | A (ours, NULL-out) | 18,335 | 111.85 | 162.58 | 1.00x |
| 100,000 | matched (>= 8.0) | B (prototype, coerce) | 18,335 | 106.16 | 155.32 | 0.95x |
| 100,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 14,065 | 104.79 | 149.56 | 1.00x |
| 100,000 | selectivity (<= 1.5) | B (prototype, coerce) | 19,074 | 105.79 | 151.81 | 1.01x |
| 1,000,000 | matched (>= 8.0) | A (ours, NULL-out) | 183,755 | 1313.85 | 1399.53 | 1.00x |
| 1,000,000 | matched (>= 8.0) | B (prototype, coerce) | 183,755 | 1284.20 | 1317.96 | 0.98x |
| 1,000,000 | selectivity (<= 1.5) | A (ours, NULL-out) | 139,421 | 1170.27 | 1214.99 | 1.00x |
| 1,000,000 | selectivity (<= 1.5) | B (prototype, coerce) | 189,274 | 1262.16 | 1358.69 | 1.08x |

### Verdict

**(a) At matched selectivity (`>= 8.0`, identical row sets), VARIANT B is NOT measurably slower than VARIANT A.** With the matched set held byte-identical (coerced 1.0/0.0 booleans can't satisfy `>= 8.0`, so both variants return only genuine numerics), B/A median ratios land at 0.95x–0.98x across 10,000, 100,000, 1,000,000 — i.e. within run-to-run noise. The two extra `IN`/equality arms in B's CASE add no observable per-row cost over A's NULL-out shape; both are dominated by the same `CAST(... AS FLOAT)` on the non-coerced majority and the same `O(N)` scan of the unindexed JSON `value` column. **The CASE shape itself is not the cost driver.** 

**(b) The real cost of coercion is selectivity, not CASE evaluation.** At `<= 1.5`, B folds `true`->1.0 and `false`->0.0 into the matched set while A NULLs them out, so B returns A's rows PLUS every boolean-valued feedback row (B_extra = 506 @ 10,000, 5009 @ 100,000, 49853 @ 1,000,000). That inflates B's matched set and pushes its median to 1.01x–1.08x of A in this scenario — the extra latency tracks the extra downstream rows the coercion admits, not the comparison operator or the CASE branches. 

**Bottom line for the prototype decision:** coercing booleans to 1.0/0.0 does *not* cost more at query time per se (the CASE shapes are equal-cost at matched selectivity); it costs more only when those coerced booleans actually fall inside the query's numeric range and become extra returned rows. Whether that is a regression depends entirely on whether surfacing boolean feedback as 1.0/0.0 in numeric filters is desired semantics — it is not a CASE-evaluation penalty.

_Dialect note: SQLite here. MySQL uses `CAST AS DOUBLE` (vs SQLite `FLOAT`), but the A-vs-B CASE-shape conclusion — equal cost at matched selectivity, divergence only via selectivity — is dialect-independent._
