# Workspace Permissions

Workspace-scoped permissions provide convenient access control for all
resources within a workspace when [authentication](/self-hosting/security/basic-http-auth)
is enabled. Workspace authorization is expressed through MLflow's role system —
see <Link to="/self-hosting/security/role-based-access-control">Role-Based Access Control</Link>
for the full model and API reference. This page covers the workspace-scoped
specifics.

## Overview

A workspace-level grant is a permission on the special
`(resource_type='workspace', resource_pattern='*')` slot under a role. It
applies to **every resource type** in that workspace (experiments, registered
models, prompts, AI Gateway resources). Two levels are grantable:

- `USE` — workspace **member**. Read every resource in the workspace; create
  new experiments and registered models.
- `MANAGE` — workspace **admin**. Full authority within the workspace,
  including granting permissions to other users via roles in this workspace.
  Cannot perform system-wide operations like deleting users.

Workspace-level grants act as a **fallback**: if a user has no explicit
resource-level permission for a resource, MLflow falls back to the workspace
permission for that resource's workspace.

### Permission resolution order

1. **Admin check** — `is_admin = true` short-circuits to allow.
2. **Explicit resource permission** — a role-derived grant on the specific
   `(resource_type, resource_id)`.
3. **Workspace permission** — a role-derived grant on `(workspace, *)`.
4. **Default permission** — only when workspaces are disabled. With workspaces
   enabled, the default is effectively **deny** unless a workspace permission
   (or explicit resource permission) grants access. For the reserved `default`
   workspace, you can opt in to inheritance via `grant_default_workspace_access`.

### Example scenarios

1. **Workspace-level grant.** Alice holds the `team-a/user` role.
   - Alice can read all experiments, registered models, and prompts in `team-a`.
2. **Resource-level exception.** Alice additionally has `NO_PERMISSIONS` on
   experiment `exp-123` in `team-a` (via a direct permission or a more specific
   role).
   - Alice can read everything in `team-a` except `exp-123`.
3. **Direct resource grant without workspace access.** Bob has `EDIT` permission
   on experiment `exp-456` in `team-a`, but no role with workspace-level access.
   - Bob can access `exp-456` only. The workspace `team-a` does not appear in
     Bob's `mlflow.list_workspaces()` results.

## Default roles

When `MLFLOW_RBAC_SEED_DEFAULT_ROLES` is on (it is, by default), MLflow seeds
two roles into every newly created workspace:

| Role    | Permission               | Intent           |
| ------- | ------------------------ | ---------------- |
| `admin` | `(workspace, *, MANAGE)` | Workspace admin  |
| `user`  | `(workspace, *, USE)`    | Workspace member |

The user who creates a workspace is automatically assigned its seeded `admin`
role. See <Link to="/self-hosting/security/role-based-access-control#default-roles">Default roles</Link>
for how to disable this seeding.

## Granting workspace access

Granting workspace-level access means assigning a role that carries a
`(workspace, *, USE)` or `(workspace, *, MANAGE)` permission. The default
seeded `user` and `admin` roles cover the common cases.

### Prerequisites

- Workspaces enabled (`--enable-workspaces`)
- Authentication enabled (`--app-name basic-auth`)
- Either super-admin privileges, or an existing workspace-admin role on the
  target workspace (so you can manage role assignments there)

### Make a user a workspace member

In the admin UI: `/admin` → Users → click the user → **Edit access** → in the
_Role assignments_ section, add `<workspace>/user` → review → apply.

Programmatically:

```python
from mlflow.server import get_app_client

auth_client = get_app_client("basic-auth", tracking_uri="http://localhost:5000")

user_role = auth_client.get_role_by_name(workspace="team-a", name="user")
auth_client.assign_role(username="alice", role_id=user_role.id)
```

### Make a user a workspace admin

```python
admin_role = auth_client.get_role_by_name(workspace="team-a", name="admin")
auth_client.assign_role(username="alice", role_id=admin_role.id)
```

### Revoke workspace access

In the admin UI: `/admin` → Users → click the user → **Edit access** → in the
_Role assignments_ section, remove the workspace's `user` or `admin` role.

Programmatically:

```python
auth_client.unassign_role(username="alice", role_id=user_role.id)
```

## Workspace visibility in `list_workspaces()`

`mlflow.list_workspaces()` returns workspaces based on workspace-level access:

- A user with a role that grants `(workspace, *, …)` on a workspace sees that
  workspace in the list.
- Direct resource permissions without a workspace-level grant do **not** make a
  workspace visible. The user can still access the specific resource directly.
- Super admins see every workspace.
- If `grant_default_workspace_access=true`, the reserved `default` workspace
  is also visible (and inherits `default_permission`).

```python
import os
import mlflow

mlflow.set_tracking_uri("http://localhost:5000")

# alice holds the team-a/user role
os.environ["MLFLOW_TRACKING_USERNAME"] = "alice"
os.environ["MLFLOW_TRACKING_PASSWORD"] = "password"
print([w.name for w in mlflow.list_workspaces()])  # ["team-a"]

# bob holds only a per-resource grant on exp-123 in team-a
os.environ["MLFLOW_TRACKING_USERNAME"] = "bob"
os.environ["MLFLOW_TRACKING_PASSWORD"] = "password"
print([w.name for w in mlflow.list_workspaces()])  # []

# but bob can still access the specific experiment if he knows it exists
mlflow.set_workspace("team-a")
mlflow.get_experiment("123")  # works
```

## Troubleshooting

### Permission denied errors

If users hit permission-denied errors:

1. Verify the user holds a role with workspace-level access:

   ```python
   roles = auth_client.list_user_roles("alice")
   for r in roles:
       print(r.workspace, r.name)
   ```

2. Check whether a more specific direct permission denies access (`NO_PERMISSIONS`
   on the resource).
3. Verify the workspace exists and the role is assigned in the right workspace.

### Workspace not visible in `list_workspaces()`

The user needs a role granting workspace-level access in that workspace —
direct per-resource grants don't suffice. Assign the seeded `user` or `admin`
role for the workspace, or create a custom role that carries a `(workspace, *, …)`
permission.
