# Span Log Levels

Span log levels classify each span by severity, modeled on Python's standard `logging` module. They let you scan a busy trace at a glance: at the default threshold the trace renders unchanged, but bumping the **Minimum log level** filter in the UI hides noise (chain plumbing, output parsing) so you see only the user-visible operations — requests, responses, tool calls.

The five levels share names with Python's `logging` module:

| Level      | Typical use                                      |
| ---------- | ------------------------------------------------ |
| `DEBUG`    | Internal/glue work — chain steps, parsers, etc.  |
| `INFO`     | User-visible operations — LLM calls, tool calls. |
| `WARNING`  | Recoverable issues you'd want to see in prod.    |
| `ERROR`    | Failures.                                        |
| `CRITICAL` | Hard stops worth paging on.                      |

## Setting the level on a span

You can pass a level when you create the span. The kwarg accepts the enum value or its name.

<TabItem value="python" label="Python" default>

```python
import mlflow
from mlflow.entities import SpanLogLevel


@mlflow.trace(log_level=SpanLogLevel.INFO)
def answer(query: str) -> str:
    docs = retrieve(query)
    return llm.generate(query, docs)


@mlflow.trace(log_level="DEBUG")  # string form
def retrieve(query: str): ...
```

You can also set the level on a `LiveSpan` after creation:

```python
with mlflow.start_span("foo") as span:
    span.set_log_level("WARNING")
```

## Defaults

Every span — whether you create it manually or it's created by an autolog integration (`mlflow.openai.autolog()`, `mlflow.langchain.autolog()`, etc.) — gets a default level based on its span type, so the verbosity filter works out of the box without you annotating anything. The default is applied in the span constructor; passing an explicit `log_level` kwarg overrides it.

| Span type                                              | Default |
| ------------------------------------------------------ | ------- |
| `LLM`, `CHAT_MODEL`, `TOOL`, `RETRIEVER`, `AGENT`, `EMBEDDING` | `INFO`  |
| Everything else (`CHAIN`, `PARSER`, `RERANKER`, `MEMORY`, `WORKFLOW`, `TASK`, `EVALUATOR`, `UNKNOWN`, custom types) | `DEBUG` |

So at the default UI threshold (DEBUG) you see everything, at INFO you see the user-visible operations and lose the chain plumbing, and at WARNING+ you only see spans you've explicitly marked above that level.

### Spans with an exception event are bumped to ERROR

If a span records an exception event — via the `@mlflow.trace` decorator catching an unhandled exception, an explicit `span.record_exception(...)` call, or a tracer-level error handler — MLflow automatically promotes the span's level to `ERROR` (unless it was already higher). This means a span that defaulted to `DEBUG` won't disappear from the trace explorer just because a user raised the verbosity filter past `INFO` — failures stay visible.

A user-set `CRITICAL` is preserved; the bump only fires when the current level is below `ERROR`.

## The UI filter

Open a trace in the MLflow UI and click **Filter** above the timeline tree. The popover has a **Minimum log level** slider (with a "?" tooltip explaining the rules):

- **Debug** (default) — show every span.
- **Info** — hide `DEBUG` spans (most chain plumbing disappears).
- **Warning / Error / Critical** — only spans explicitly marked at or above the threshold.

The existing **Show all parent spans** and **Show exceptions** toggles still apply: they override the level filter so you don't lose context for matched children or for spans that recorded an exception.

## Backwards compatibility

Spans persisted before MLflow 3.13 (or spans from third-party tracers that don't stamp a level) have no `mlflow.spanLogLevel` attribute. The UI treats them as `DEBUG` so they remain visible at the default threshold and don't disappear unexpectedly when you bump the filter. If you raise the threshold to `INFO` or higher, those unclassified spans will be filtered out — at which point you can opt in by stamping levels manually with `set_log_level`.
