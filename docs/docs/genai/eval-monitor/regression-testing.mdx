# Self-Evolving Test Suite

Have you ever fixed one thing in your agent only to break something else? You shorten a verbose answer, a stakeholder asks for another tweak, and the new fix quietly clobbers the answer that used to work. Every fix can revive an old bug, and coding agents make it worse: they rewrite your whole agent in seconds while your verification stays manual. Your fixes end up competing instead of compounding, and you never quite feel ready to ship.

A **self-evolving test suite** breaks that cycle by putting MLflow inside the loop you already run. You keep vibe-checking your agent and handing feedback to your coding agent, but now every piece of feedback becomes a permanent test, and every later change runs against every failure you have ever recorded. Old fixes stay green, fixes compound instead of competing, and by the time the agent is ready to ship you have a real regression suite. When the suite is green, you are ready to ship.

This tutorial builds that suite end to end on a small support agent. You will trace the agent, lock a behavior in as a test, make a well-meaning change that silently breaks it, watch the suite catch the regression, and then gate the suite in CI. The finished product is a suite that makes regressions impossible to miss:

<ImageBox src="/images/mlflow-3/eval-monitor/self-evolving-eval-runs.png" alt="MLflow Evaluation runs page comparing two runs of a regression suite, where the is_concise score drops from 1 to 0 after a change" width="100%" />

## How the loop works

The best suites are not written up front, they grow from real failures. You keep doing exactly what you already do, vibe-checking your agent and telling your coding agent what went wrong, and MLflow turns each piece of feedback into a test underneath you.

## Prerequisites

Install MLflow 3.14 or newer (when `@mlflow.test` shipped) along with the OpenAI SDK:

```bash
pip install --upgrade "mlflow>=3.14" openai
```

Set your API key, pin the judge model so every run grades the same way, and point at a local tracking server:

```bash
export OPENAI_API_KEY="your-api-key"
export MLFLOW_GENAI_JUDGE_DEFAULT_MODEL="openai:/gpt-5-mini"
export MLFLOW_TRACKING_URI="http://localhost:5000"
```

Then start that server in a separate terminal (you installed MLflow above) so your traces and runs are recorded and reviewable:

```bash
mlflow server
```

## Step 1: Build an agent you can trace

Create `agent.py`. It is a tiny customer support assistant: a single OpenAI call with a system prompt. The one MLflow-specific line is <APILink fn="mlflow.openai.autolog" />, which traces every call automatically.

```python
# agent.py
import mlflow
import openai

mlflow.openai.autolog()
mlflow.set_experiment("support-agent")

SYSTEM_PROMPT = (
    "You are a customer support assistant. "
    "Our support team is available 9 AM to 5 PM Eastern Time, Monday through Friday. "
    "To reset a password, click 'Forgot password' on the sign-in page. "
    "Answer the customer's question concisely, in one short sentence."
)


def invoke(question: str) -> str:
    client = openai.OpenAI()
    completion = client.chat.completions.create(
        model="gpt-5.5",
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": question},
        ],
    )
    return completion.choices[0].message.content
```

Try it:

```python
import agent

print(agent.invoke("What are your support hours?"))
# Our support hours are Monday through Friday, 9 AM to 5 PM Eastern Time.
```

Open the MLflow UI and select the **support-agent** experiment. Every call shows up under the **Traces** tab with its inputs, outputs, and token counts, so when an answer is wrong you inspect the real trace instead of guessing. This is the foundation everything else builds on.

## Step 2: Turn a behavior into a test

The answer above is concise, exactly what you want. Lock that in so it can never silently regress.

A test is an ordinary `pytest` function. You mark it with `@mlflow.test`, call <APILink fn="mlflow.genai.evaluate" /> with your agent and the scorers that encode the behavior you care about, then assert on the result. By the end of this step your project looks like this:

```
support-agent/
  agent.py
  pyproject.toml
  tests/regression/
    test_support_agent.py
```

Keep `agent.py` at the project root and create the `tests/regression/` directory for the test. The test imports the agent directly. Here the two [`Guidelines`](/genai/eval-monitor/scorers/llm-judge/guidelines) scorers are LLM judges that check the answer in plain language:

```python
# tests/regression/test_support_agent.py
import mlflow
from mlflow.genai.scorers import Guidelines

import agent


@mlflow.test
def test_support_hours_is_concise():
    result = mlflow.genai.evaluate(
        predict_fn=agent.invoke,
        data=[{"inputs": {"question": "What are your support hours?"}}],
        scorers=[
            Guidelines(
                name="is_concise", guidelines="The answer must be concise: at most three sentences."
            ),
            Guidelines(name="is_english", guidelines="The answer must be written in English."),
        ],
    )
    assert result.passed, result.reason
```

`@mlflow.test` runs through a pytest plugin. Enable it once in your `pyproject.toml` so every `pytest` run picks it up. The `pythonpath` entry puts the project root on the import path so the test can `import agent` under the bare `pytest` command:

```toml
[tool.pytest.ini_options]
addopts = ["-p", "mlflow.pytest.plugin"]
pythonpath = ["."]
```

For a one-off run without editing `pyproject.toml`, pass `pytest -p mlflow.pytest.plugin` instead. Run it like any other pytest suite:

```bash
pytest tests/regression/test_support_agent.py
```

`result.passed` is `True` only when every scorer passed for every row. `result.reason` names the failing scorers and their rationales, so a failed assertion tells you _why_ without a second run. The whole session is recorded as a single MLflow run; open the **Evaluation runs** page and select it to inspect each test case, its conversation, and per-assertion results.

## Step 3: Watch a change get caught

Now a new requirement arrives: customers find the agent terse, so you are asked to make it friendlier. First, capture that as its own test, so "friendly" is also a behavior you guard:

```python
@mlflow.test
def test_greeting_is_friendly():
    result = mlflow.genai.evaluate(
        predict_fn=agent.invoke,
        data=[{"inputs": {"question": "Hi, I need some help."}}],
        scorers=[
            Guidelines(
                name="is_friendly",
                guidelines="The answer must greet the customer warmly and politely.",
            ),
        ],
    )
    assert result.passed, result.reason
```

Then make the change. Edit the system prompt in `agent.py` to be warm and thorough:

```python
SYSTEM_PROMPT = (
    "You are a customer support assistant. "
    "Our support team is available 9 AM to 5 PM Eastern Time, Monday through Friday. "
    "To reset a password, click 'Forgot password' on the sign-in page. "
    "Always greet the customer warmly, thank them for reaching out, explain your answer "
    "thoroughly with background, and end with a friendly offer to help further."
)
```

Run the whole suite:

```bash
pytest tests/regression
```

The greeting test passes, but `test_support_hours_is_concise` now **fails**. Making the agent friendlier made it verbose, which broke the conciseness you locked in two steps ago. Without the suite, nobody would have noticed until a customer complained. Open the failing case in the UI to see exactly what happened: the conversation, and the `is_concise` assertion flipped to **Fail** with the judge's rationale.

<ImageBox src="/images/mlflow-3/eval-monitor/self-evolving-regression-failure.png" alt="A failing test case: a verbose support reply, where the is_english assertion passes but the is_concise assertion fails" width="50%" />

This is the doom loop made visible. The fix for friendliness competed with the fix for conciseness.

## Step 4: Make the fixes compound

Reconcile the two requirements. Edit the system prompt once more so the agent is both friendly _and_ concise:

```python
SYSTEM_PROMPT = (
    "You are a customer support assistant. "
    "Our support team is available 9 AM to 5 PM Eastern Time, Monday through Friday. "
    "To reset a password, click 'Forgot password' on the sign-in page. "
    "Open with a brief friendly greeting, then answer the question in a single short sentence."
)
```

Run the suite again:

```bash
pytest tests/regression
```

Everything is green. The same change that satisfies the friendliness requirement now keeps the conciseness one too. Open the previously failing case and both assertions pass:

<ImageBox src="/images/mlflow-3/eval-monitor/self-evolving-test-passed.png" alt="The same test case after the fix: a concise, friendly reply where both the is_concise and is_english assertions pass" width="50%" />

That is the difference between the old loop and the new one. In the old loop your fixes compete and you move backwards. Here every change runs against everything you have ever locked in, so fixes compound and the suite only grows greener. The **Evaluation runs** page makes the trend obvious (the comparison at the top of this page): `is_concise` dropped to 0 on the friendlier rewrite and is back to 1 once both behaviors hold.

## Step 5: Gate the suite in CI

Because these are plain pytest tests, running them in CI is just running pytest. Point `MLFLOW_TRACKING_URI` at a persistent tracking server (or a Databricks workspace) so the run and its traces are recorded and reviewable after the job finishes, and pin the judge model so every run grades the same way.

```yaml
# .github/workflows/agent-regression.yml
name: Agent regression tests
on: [pull_request]

jobs:
  regression:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install "mlflow>=3.14" pytest openai
      - name: Run regression tests
        env:
          MLFLOW_TRACKING_URI: ${{ secrets.MLFLOW_TRACKING_URI }}
          # Judge model for the LLM-judge scorers.
          MLFLOW_GENAI_JUDGE_DEFAULT_MODEL: "openai:/gpt-5-mini"
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: pytest -p mlflow.pytest.plugin tests/regression
```

A failing assertion fails the pytest job, which fails the check, which blocks the pull request, exactly like a unit test. The `result.reason` in the pytest output tells you which scorer flagged the output and why, and the recorded MLflow run lets you open the full trace to debug.

### Run a large suite in parallel

Agent tests are slow: each one runs your real agent and its LLM-judge scorers. As your suite grows, spread the tests across processes with [pytest-xdist](https://pytest-xdist.readthedocs.io/). Add a hook to `conftest.py` so every worker reports to a single MLflow run instead of scattering one run per worker:

```python
# conftest.py
import os
import mlflow


def pytest_configure(config):
    # Put every worker's test cases on a single MLflow run.
    if not hasattr(config, "workerinput"):  # controller only
        run = mlflow.start_run(run_name="regression-suite")
        mlflow.end_run()
        os.environ["MLFLOW_RUN_ID"] = run.info.run_id
```

Then run with `pytest -n auto` (one worker per CPU core) or `pytest -n 4` for a fixed number.

## This is eval, just not the eval you are dreading

What you just did is evaluation, just not the version most teams dread. Traditional eval asks you to build a golden dataset, write custom judges, and align them to human experts before you get any value back. That work is real and it matters, but the upfront cost is why so many teams stall in the vibe-check loop and never start.

This inverts the order. You started evaluating from the first answer you checked, using feedback you were already going to give. No golden dataset or judge alignment on day one. Each test is a real `pytest` backed by real MLflow scoring, and later, as you learn what matters, you can layer in [built-in judges](/genai/eval-monitor/scorers/llm-judge/predefined), [custom judges](/genai/eval-monitor/scorers/llm-judge/custom-judges), and [judge alignment](/genai/eval-monitor/scorers/llm-judge/alignment). You are not skipping eval, you are bootstrapping it, so the suite grows into eval instead of eval gating your progress.
