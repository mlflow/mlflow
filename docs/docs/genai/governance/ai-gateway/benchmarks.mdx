# AI Gateway Performance Benchmarks

MLflow AI Gateway is designed to add minimal overhead when proxying requests to LLM providers. This page documents our benchmark methodology and results so you can evaluate gateway performance for your workloads.

## Methodology

Our benchmark suite measures **pure proxy overhead** — the extra latency each gateway adds on top of the upstream provider call. MLflow, LiteLLM, and Portkey are tested under identical conditions for direct comparability.

- **Benchmark client**: `aiohttp` with connection pooling (matching [LiteLLM's own benchmark methodology](https://docs.litellm.ai/docs/benchmarks))
- **Backend**: Fake OpenAI-compatible server returning canned responses with configurable delay (`FAKE_RESPONSE_DELAY_MS`, default 50ms)
- **Measurement**: Client-side `time.perf_counter()` timestamps (high-resolution monotonic clock)
- **Warmup**: 50 requests discarded before each timed run to establish connection pools and warm caches
- **Environment**: All proxies run on the same machine, same number of workers, hitting the same fake backend
- **Competitors**: [LiteLLM Proxy](https://docs.litellm.ai/docs/) (Python, PostgreSQL) and [Portkey AI Gateway](https://github.com/Portkey-AI/gateway) (Node.js, stateless)

By controlling for network and provider latency (everything runs on `127.0.0.1`), the benchmark isolates what each gateway adds to the request path.

## Results

All results: MacBook Pro (Apple Silicon), 4 workers, 50 concurrent, 2000 requests/run, 3 runs.

### Head-to-head (SQLite, no tracking, 50ms delay)

With config caching enabled (the default since [#21660](https://github.com/mlflow/mlflow/pull/21660)) and usage tracking off — the "core proxy overhead" comparison:

| Metric | MLflow AI Gateway | LiteLLM | Portkey |
| --- | --- | --- | --- |
| **P50 latency** | 56.7 ms | 76.8 ms | **52.5 ms** |
| **P95 latency** | 77.1 ms | 104.0 ms | **55.8 ms** |
| **P99 latency** | 110.8 ms | 256.7 ms | **59.0 ms** |
| **Throughput** | **818 rps** | 616 rps | **941 rps** |

### Pure proxy overhead (zero delay)

With `FAKE_RESPONSE_DELAY_MS=0`, measuring raw proxy speed:

| Metric | MLflow AI Gateway | LiteLLM | Portkey |
| --- | --- | --- | --- |
| **P50 latency** | 13.6 ms | 40.1 ms | **8.0 ms** |
| **P99 latency** | 64.5 ms | 232.5 ms | **18.1 ms** |
| **Throughput** | **3,306 rps** | 1,181 rps | **5,575 rps** |

MLflow's core proxy path adds ~14ms of overhead vs LiteLLM's ~40ms — **2.9x faster** when measuring raw proxy speed. Portkey, being a stateless Node.js proxy with no database, achieves the lowest overhead at ~8ms.

### Full-stack with PostgreSQL and tracking

The production configuration: PostgreSQL backend, usage tracking / spend tracking enabled for both MLflow and LiteLLM. Portkey's OSS version has no tracking or database.

| Metric | MLflow AI Gateway | LiteLLM | Portkey |
| --- | --- | --- | --- |
| **P50 latency** | **54.8 ms** | 103.0 ms | **52.7 ms** |
| **P95 latency** | **72.1 ms** | 154.7 ms | **57.3 ms** |
| **P99 latency** | **102.4 ms** | 298.8 ms | **68.2 ms** |
| **Throughput** | **852 rps** | 461 rps | **932 rps** |

MLflow with full tracing enabled (852 rps) is faster than LiteLLM without any tracking (602 rps), and within 9% of stateless Portkey (932 rps).

:::note
The full-stack tracking-ON results include performance optimizations that are under review and not yet merged to `master`. On current `master`, MLflow achieves ~77 rps with tracking enabled. See the [benchmark README](https://github.com/mlflow/mlflow/tree/master/benchmarks/gateway/README.md) for the full optimization journey (77 → 852 rps).
:::

:::note
Results were collected on a MacBook Pro (Apple Silicon). Run the benchmark on your own hardware for environment-specific numbers.
:::

## Bottleneck Analysis

The MLflow AI Gateway has two configurable bottlenecks that affect performance:

### 1. Endpoint config resolution

MLflow treats the database as the source of truth for endpoint configuration, resolving it with 3-5 SQL queries per request. LiteLLM loads config from YAML at startup and keeps it in memory.

**Config caching** (enabled by default via [#21660](https://github.com/mlflow/mlflow/pull/21660)) caches the result of `get_endpoint_config()` after the first DB lookup, eliminating per-request SQL queries and giving a **3.4x throughput improvement**.

### 2. Usage tracking / tracing

When usage tracking is enabled, every request is wrapped with `mlflow.trace()` for observability. This adds span creation, metadata recording, and async trace persistence. Several optimizations have been developed to minimize this overhead:

- **DB contention fix**: Consolidate span + trace writes into a single transaction (eliminates `UniqueViolation` and `DeadlockDetected` errors)
- **Pure ASGI middleware**: Convert Starlette's `BaseHTTPMiddleware` to raw ASGI (eliminates 30ms event loop overhead per request)
- **Batch span processor**: Decouple trace export from the request path via OTel's `BatchSpanProcessor` (exports in background thread instead of inline)

### Impact breakdown

The following table shows how each bottleneck affects throughput (PostgreSQL, 4 workers, 50 concurrent):

| Configuration | MLflow (rps) | LiteLLM (rps) |
| --- | --- | --- |
| No cache, usage tracking ON (worst case) | 14 | 430 |
| Cache ON, usage tracking ON (current `master`) | ~77 | 521 |
| Cache ON, usage tracking ON (with optimizations) | **852** | 461 |
| **Cache ON, usage tracking OFF** | **841** | **539** |

With all optimizations applied, usage tracking adds only ~4% overhead (852 vs 891 rps without tracking). Both bottlenecks are independent — config caching is already shipped; tracking optimizations are under review.

### Perspective: does proxy overhead matter?

With a real provider adding 200-2000ms of inference time, a 10-50ms proxy overhead is 0.5-5% of total latency. The benchmarks are most useful for identifying optimization opportunities and comparing architectures under controlled conditions.

## Running Benchmarks Yourself

The benchmark suite lives in [`benchmarks/gateway/`](https://github.com/mlflow/mlflow/tree/master/benchmarks/gateway).

```bash
# Install dependencies
pip install mlflow[gateway] 'litellm[proxy]'

# Head-to-head comparison (MLflow vs LiteLLM vs Portkey)
cd benchmarks/gateway
GATEWAY_WORKERS=4 REQUESTS=2000 MAX_CONCURRENT=50 RUNS=3 bash run_comparison.sh

# MLflow-only tracking server benchmark
bash run_tracking_server_benchmark.sh

# Full-stack comparison with PostgreSQL (requires Docker)
bash run_full_stack_comparison.sh
```

See the [benchmark README](https://github.com/mlflow/mlflow/tree/master/benchmarks/gateway/README.md) for full configuration options, deployment guides, and detailed methodology.
